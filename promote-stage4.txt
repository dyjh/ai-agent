任务 1：规范 MCP 配置结构
==================================================

确保支持如下配置格式。

config/mcp.servers.yaml:

```yaml
servers:
  - id: filesystem
    name: filesystem
    enabled: true
    transport: stdio
    command: node
    args:
      - ./mcp/filesystem/index.js
    cwd: ./mcp/filesystem
    env:
      ROOT_DIR: ./workspace

  - id: local-http-tools
    name: local-http-tools
    enabled: true
    transport: http
    url: http://localhost:3001/mcp
    headers:
      Authorization: ${MCP_LOCAL_TOKEN}
````

config/mcp.tool-policies.yaml:

```yaml
tools:
  mcp.filesystem.read_file:
    effects:
      - fs.read
    approval: auto

  mcp.filesystem.write_file:
    effects:
      - fs.write
    approval: require

  mcp.github.create_issue:
    effects:
      - network.post
    approval: require
```

要求：

* 支持 env expansion
* disabled server 不可被调用
* 缺失配置给出明确错误
* 不要打印 secret 值
* header/env 中的 secret 不应进入日志
* 支持 stdio/http 两种 transport 的差异化字段校验
* 如果 transport=stdio，则必须有 command
* 如果 transport=http，则必须有 url

任务 2：定义 MCP 运行时接口
=================

请定义或完善以下接口。

```go
type MCPToolSchema struct {
    Name        string         `json:"name"`
    Description string         `json:"description,omitempty"`
    InputSchema map[string]any `json:"input_schema,omitempty"`
    Metadata    map[string]any `json:"metadata,omitempty"`
}

type MCPToolResult struct {
    Content  any            `json:"content,omitempty"`
    Structured map[string]any `json:"structured,omitempty"`
    Raw      map[string]any `json:"raw,omitempty"`
}

type MCPTransport interface {
    Start(ctx context.Context) error
    ListTools(ctx context.Context) ([]MCPToolSchema, error)
    CallTool(ctx context.Context, name string, input map[string]any) (*MCPToolResult, error)
    Close(ctx context.Context) error
}

type MCPClient interface {
    ListTools(ctx context.Context, serverID string) ([]MCPToolSchema, error)
    CallTool(ctx context.Context, serverID string, toolName string, input map[string]any) (*MCPToolResult, error)
    Health(ctx context.Context, serverID string) error
}
```

要求：

* transport 不应该直接暴露给 handler
* client 负责按 server_id 选择 transport
* manager 负责配置、状态、refresh、policy overlay
* transport 应可 mock，便于测试
* transport 生命周期要可控

任务 6：实现 mcp.call_tool Tool
==========================

将 `mcp.call_tool` 从占位实现改成真实执行器。

ToolSpec 目标格式：

```json
{
  "id": "mcp.call_tool",
  "provider": "mcp",
  "name": "call_mcp_tool",
  "description": "Call a tool exposed by a configured MCP server",
  "default_effects": ["unknown.effect"],
  "input_schema": {
    "type": "object",
    "properties": {
      "server_id": { "type": "string" },
      "tool_name": { "type": "string" },
      "arguments": { "type": "object" }
    },
    "required": ["server_id", "tool_name"]
  }
}
```

执行链路要求：

* Agent 生成 ToolProposal
* ToolRouter 接收 mcp.call_tool proposal
* EffectInference 先查本地 override
* 再结合 server schema
* 最终给出 effects / risk / confidence
* PolicyEngine 决定 auto / approval
* 需要审批时创建 approval.input_snapshot：

  * server_id
  * tool_name
  * arguments
* 审批通过后才真正调用 MCPClient.CallTool
* Executor 只执行批准过的 input_snapshot

不要出现这种错误路径：

* Eino Tool 直接 CallTool
* handler 直接 CallTool
* approval 前已实际调用远端 tool
