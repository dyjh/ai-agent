任务 1：实现 code.run_tests
==================================================

新增 Tool：

```text
code.run_tests

目标：

让 Agent 能基于项目类型和已检测到的测试命令运行测试，并把测试结果结构化返回给 workflow。

建议输入：

type CodeRunTestsInput struct {
    Workspace       string   `json:"workspace"`
    Command         string   `json:"command,omitempty"`
    Args            []string `json:"args,omitempty"`
    TimeoutSeconds  int      `json:"timeout_seconds,omitempty"`
    MaxOutputBytes  int64    `json:"max_output_bytes,omitempty"`
    UseDetected     bool     `json:"use_detected,omitempty"`
    TestNamePattern string   `json:"test_name_pattern,omitempty"`
}

建议输出：

type CodeRunTestsResult struct {
    Workspace       string   `json:"workspace"`
    Command         string   `json:"command"`
    ExitCode        int      `json:"exit_code"`
    Passed          bool     `json:"passed"`
    DurationMS      int64    `json:"duration_ms"`
    Stdout          string   `json:"stdout"`
    Stderr          string   `json:"stderr"`
    Truncated       bool     `json:"truncated"`
    DetectedCommand bool     `json:"detected_command"`
}

要求：

如果 input.Command 为空且 UseDetected=true，则调用现有 code.detect_test_command 逻辑。
如果检测到多个测试命令，优先选择最适合当前项目的一个：
Go: go test ./...
Node: npm test / pnpm test / yarn test
Python: pytest
Rust: cargo test
Makefile: make test
测试命令执行必须走受控 executor，不要直接从 LLM 文本进入 exec。
timeout 必须生效。
max output bytes 必须生效。
stdout/stderr 必须脱敏。
测试命令如果明显会安装依赖、修改文件、修改 lockfile，必须进入审批。
普通只读测试可以自动执行。
执行结果必须写入 AgentEvent / JSONL / run step。
不要把完整巨量测试输出无限写入日志。

Effect 建议：

code.test
process.read
fs.read

如果测试命令包含写入、安装、生成文件，则 effect inference 应增加：

fs.write
package.install
code.modify
unknown.effect

并要求审批。

任务 3：实现 code.fix_test_failure_loop

新增高层工具或 workflow capability：

code.fix_test_failure_loop

注意：它可以是 Tool，也可以是 Workflow 内部能力。若作为 Tool，仍不能直接写文件，必须通过 propose_patch/apply_patch 审批链路。

目标：

让 Agent 能在有限次数内完成：

运行测试
解析失败
搜索相关代码
读取相关文件
生成修复 patch
请求审批
apply patch
再运行测试
达到通过或 max iterations 后停止

建议配置：

code:
  max_fix_iterations: 3
  max_test_runtime_seconds: 300
  max_test_output_bytes: 200000
  allow_auto_run_readonly_tests: true

建议结构：

type CodeFixLoopInput struct {
    Workspace      string `json:"workspace"`
    TestCommand    string `json:"test_command,omitempty"`
    MaxIterations  int    `json:"max_iterations,omitempty"`
    StopOnApproval bool   `json:"stop_on_approval,omitempty"`
}

type CodeFixLoopResult struct {
    Iterations       int      `json:"iterations"`
    FinalPassed      bool     `json:"final_passed"`
    AppliedPatches   []string `json:"applied_patches"`
    RemainingFailures []CodeTestFailure `json:"remaining_failures,omitempty"`
    Summary          string   `json:"summary"`
}

要求：

每次修改仍必须走 code.propose_patch -> approval -> code.apply_patch。
不能自动 apply patch。
StopOnApproval=true 时，生成 patch 后暂停，让用户确认。
approval 后 workflow resume 再继续下一轮测试。
每轮必须落 run step。
必须有 max iterations 上限。
达到上限后停止，并说明未完成原因。
如果测试命令本身需要审批，先请求测试命令审批。
不要无限循环。

实现建议：

先做 workflow-level loop，而不是一口气做复杂 autonomous repair。
可以先支持“生成下一步 patch proposal 并暂停审批”。
不要求本阶段让模型百分百修好测试，但必须把闭环框架跑通。
