==================================================
绝对不能破坏的既有原则
==================================================

1. 不要引入租户、多用户、RBAC、云端 SaaS 假设。
2. 本系统仍是单用户本地部署。
3. 模型不能直接操作外部世界。
4. LLM / Eino Tool 只能生成结构化 ToolProposal。
5. 所有外部动作必须继续经过：

   ToolProposal
   -> EffectInference
   -> PolicyEngine
   -> ApprovalCenter（需要时）
   -> Executor

6. 只读高置信操作可以自动执行。
7. 写操作、安装依赖、敏感读取、危险操作、网络写入、unknown.effect、低置信操作必须审批。
8. approval 后只能执行 approval.input_snapshot。
9. Resume 时不能重新规划替换已审批 input_snapshot。
10. 不要让 HTTP handler / WebSocket handler / CLI 直接绕过 workflow 或 ToolRouter。
11. 不要把 secret、token、cookie、私钥、.env 正文写进：
    - messages
    - JSONL
    - agent_events
    - Markdown memory
    - Qdrant payload
    - Swagger 示例
12. Qdrant 仍只是索引，不是事实源。
13. Markdown Memory 仍是长期记忆事实源。
14. 如果新增或修改 API，必须补 swag 注释并重新生成 Swagger/OpenAPI。
15. README、task.json、Swagger 不允许领先于实际实现。
16. 不要回退已有 KB / Skill / MCP / Workflow / Sandbox / Swagger 能力。

==================================================
整体目标：全能型 Agent 完成标准
==================================================

最终希望系统能稳定完成四类任务：

一、聊天：
- 能自然多轮对话
- 能使用长期记忆
- 能区分用户偏好、项目记忆、操作经验和临时上下文
- 不乱记敏感信息
- 用户可以查看、编辑、删除记忆

二、问答：
- 能基于本地知识库可靠回答
- 支持引用来源
- 支持增量索引
- 支持混合检索、rerank
- 支持“只基于知识库回答”
- 知识库内容不能覆盖系统规则

三、运维：
- 能执行安全的只读排查
- 写操作、高危操作需要审批
- 支持 local / SSH / Docker / Kubernetes 等 host profile
- 支持 runbook
- 支持 rollback plan
- 不泄露凭证

四、写代码：
- 能读仓库
- 能搜索代码
- 能理解依赖和测试命令
- 能生成 patch
- 审批后 apply patch
- 能运行测试
- 能根据测试失败继续修复
- 能总结 diff / commit message / PR summary

Phase F1：代码能力闭环
==================================================

这是最高优先级阶段。

目标：

让 Agent 从“能调用工具”升级为“能完成真实代码任务”。

最终流程：

用户说：“帮我实现/修复这个功能”
Agent 应该能：

1. 分析需求
2. 检查仓库结构
3. 搜索相关代码
4. 读取相关文件
5. 制定修改计划
6. 生成 patch
7. 请求用户审批
8. apply patch
9. 运行测试
10. 根据测试失败继续修复
11. 输出变更总结
12. 输出测试结果
13. 可选生成 commit message

----------------------------------------
F1.1 Code Workspace 能力
----------------------------------------

新增或完善工具：

- code.list_files
- code.read_file
- code.search_text
- code.search_symbol
- code.inspect_project
- code.detect_language
- code.detect_test_command
- code.propose_patch
- code.apply_patch
- code.run_tests
- code.explain_diff

要求：

1. `code.read_file` 普通文件只读可自动执行。
2. 读取敏感文件，例如 `.env`、private key、credentials，必须 sensitive_read 审批。
3. `code.propose_patch` 不修改文件，可自动执行。
4. `code.apply_patch` 修改文件，必须审批。
5. 审批对象是具体 diff / patch snapshot。
6. approval 后只能应用 approval.input_snapshot 中的 patch。
7. 运行测试可按策略判断：
   - 只读测试命令可自动执行
   - 安装依赖、写文件、修改 lockfile 的测试准备步骤必须审批
8. 所有代码工具调用写入 run steps 和 JSONL。

## F1.5 测试闭环

新增：

* code.run_tests
* code.parse_test_failure
* code.fix_test_failure_loop

流程：

1. Agent 运行测试。
2. 解析失败。
3. 搜索相关代码。
4. 生成修复 patch。
5. 审批后 apply。
6. 再跑测试。
7. 达到 max_fix_iterations 后停止。

配置：

```yaml
code:
  max_fix_iterations: 3
  max_test_runtime_seconds: 300
  allow_auto_run_readonly_tests: true
```

验收：

* 能完成一个简单 Go 项目的 bugfix。
* 能生成 patch 并审批后应用。
* 能运行 go test ./...。
* 测试失败后能继续尝试一次修复。
* go test ./... 通过。
* 新增代码能力 tests 通过。
