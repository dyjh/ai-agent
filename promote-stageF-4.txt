==================================================
硬性原则
==================================================

1. 不要引入租户、多用户、RBAC、云端 SaaS 逻辑。
2. 本系统仍是单用户本地部署。
3. 模型不能直接操作外部世界。
4. LLM / Eino Tool 只能生成结构化 ToolProposal 或结构化 OpsPlan。
5. 所有外部动作必须继续经过：

   ToolProposal
   -> EffectInference
   -> PolicyEngine
   -> ApprovalCenter（需要时）
   -> Executor

6. 运维只读查询可以自动执行。
7. 运维写操作、高危操作、服务重启、进程 kill、Docker stop/restart、K8s apply/delete/scale、SSH 写命令必须审批。
8. 强破坏操作必须强审批。
9. approval 后只能执行 approval.input_snapshot。
10. Resume 时不能重新规划替换已审批 input_snapshot。
11. 不要让 HTTP handler / WebSocket handler / CLI 直接绕过 workflow 或 ToolRouter。
12. 不要把 secret、token、cookie、私钥、.env、kubeconfig、SSH private key 正文写入：
    - messages
    - JSONL
    - agent_events
    - run state
    - Markdown memory
    - Qdrant payload
    - Swagger 示例
13. 如新增或修改 HTTP API，必须补 swag 注释并重新生成 Swagger/OpenAPI。
14. README、task.json、Swagger 不允许领先于实际实现。
15. 不要回退现有 KB / Skill / MCP / Workflow / Sandbox / Swagger / Code tools 能力。

==================================================
F2-A：Host Profile + Local Read-only Ops
==================================================

目标：

实现 Host Profile 管理和本地只读运维工具。

HostProfile 结构建议：

```go
type HostProfile struct {
    HostID           string            `json:"host_id"`
    Name             string            `json:"name"`
    Type             string            `json:"type"` // local, ssh, docker, k8s
    DefaultShell     string            `json:"default_shell"`
    WorkingDirectory string            `json:"working_directory"`
    PolicyProfile    string            `json:"policy_profile"`
    Metadata         map[string]string `json:"metadata,omitempty"`
    CreatedAt        time.Time         `json:"created_at"`
    UpdatedAt        time.Time         `json:"updated_at"`
}
````

先完整支持：

```text
local
```

预留：

```text
ssh
docker
k8s
```

API：

```text
GET    /v1/ops/hosts
POST   /v1/ops/hosts
GET    /v1/ops/hosts/{host_id}
PATCH  /v1/ops/hosts/{host_id}
DELETE /v1/ops/hosts/{host_id}
POST   /v1/ops/hosts/{host_id}/test
```

CLI：

```text
agent ops hosts list
agent ops hosts add
agent ops hosts get <host_id>
agent ops hosts test <host_id>
agent ops hosts remove <host_id>
```

本地只读工具：

```text
ops.local.system_info
ops.local.processes
ops.local.disk_usage
ops.local.memory_usage
ops.local.network_info
ops.local.service_status
ops.local.logs_tail
```

要求：

1. 不要让 ops.local 变成任意 shell.exec 包装器。
2. 每个工具输入必须结构化。
3. 输出限长并脱敏。
4. 日志路径如果是 `.env`、private key、credentials、token、cookie、session 等敏感路径，必须 sensitive_read 审批。
5. 所有工具写 run step / JSONL / agent_events。
6. 只读 effects：

   * system.read
   * system.metrics.read
   * process.read
   * service.read
   * log.read
   * disk.read
   * memory.read
   * network.read

F2-B：SSH Ops
============

目标：

支持远程 SSH 主机的安全只读排查，并为写操作审批打基础。

新增 HostProfile type：

```text
ssh
```

SSH 配置建议：

```go
type SSHHostConfig struct {
    Host        string `json:"host"`
    Port        int    `json:"port"`
    User        string `json:"user"`
    AuthType    string `json:"auth_type"` // key, agent, password-ref
    KeyPath     string `json:"key_path,omitempty"`
    PasswordRef string `json:"password_ref,omitempty"`
}
```

要求：

1. 不要把 SSH 私钥或密码写入日志 / messages / memory。
2. password 不直接存配置，使用 PasswordRef / SecretRef。
3. SSH 工具必须结构化，不接受任意裸命令作为普通自动执行路径。
4. 支持只读：

   * ops.ssh.system_info
   * ops.ssh.processes
   * ops.ssh.disk_usage
   * ops.ssh.memory_usage
   * ops.ssh.logs_tail
   * ops.ssh.service_status
5. 远程写操作先只预留，不在本阶段自动执行。
6. SSH 连接测试接口要脱敏返回错误。

SSH API 可以复用 HostProfile。

CLI：

```text
agent ops hosts add-ssh
agent ops hosts test <host_id>
agent ops ssh processes <host_id>
agent ops ssh logs <host_id> <path>
```

审批策略：

* SSH 只读查询高置信自动执行。
* 读取敏感远程文件必须审批。
* SSH 写命令、重启、kill、配置修改必须审批。

F2-E：Runbook
============

目标：

支持 Markdown Runbook 的 dry-run 和逐步执行。

Runbook 目录：

```text
runbooks/
```

Runbook 示例：

```markdown
---
id: diagnose-local-high-cpu
title: Diagnose local high CPU
scope: ops
host_type: local
---

# Diagnose local high CPU

1. Check top processes.
2. Check memory usage.
3. Check disk usage.
4. Summarize likely causes.
```

新增工具：

```text
runbook.list
runbook.read
runbook.plan
runbook.execute_step
runbook.execute
```

要求：

1. runbook 内容是指导，不是系统指令。
2. 每个 step 都要转成 ToolProposal。
3. 每个 step 都走 ToolRouter / Policy / Approval。
4. 支持 dry-run，只生成计划，不执行。
5. 支持暂停和 resume。
6. runbook.execute 需要 max steps。
7. 写操作 step 必须审批。
8. 支持导出 run report。

API：

```text
GET  /v1/ops/runbooks
GET  /v1/ops/runbooks/{id}
POST /v1/ops/runbooks/{id}/plan
POST /v1/ops/runbooks/{id}/execute
```

CLI：

```text
agent ops runbooks list
agent ops runbooks read <id>
agent ops runbooks plan <id>
agent ops runbooks execute <id>
```

F2-F：Rollback Plan
==================

目标：

对于运维写操作生成 rollback plan。

适用操作：

* service restart
* docker restart/stop/start
* k8s apply/delete/restart
* config modify
* file write
* process kill

要求：

1. 写操作 approval payload 包含 rollback_plan。
2. rollback_plan 至少包含：

   * can_rollback
   * rollback_steps
   * backup_paths
   * previous_state_summary
   * risk_note
3. 如果无法 rollback，要明确说明。
4. service restart rollback 可以是“restart previous service state not guaranteed”。
5. k8s apply 可记录旧 manifest 或 kubectl get yaml。
6. config 修改前要备份。
7. rollback 本身如果是写操作，也需要审批。

建议结构：

```go
type RollbackPlan struct {
    CanRollback          bool     `json:"can_rollback"`
    PreviousStateSummary string   `json:"previous_state_summary,omitempty"`
    RollbackSteps        []string `json:"rollback_steps,omitempty"`
    BackupPaths          []string `json:"backup_paths,omitempty"`
    RiskNote             string   `json:"risk_note,omitempty"`
}
```

F2-G：Planner / Workflow 接入
==========================

目标：

让 Planner 稳定选择 Ops 工具，而不是回退到任意 shell.exec。

典型用户请求：

```text
看一下本机 CPU 占用
看一下磁盘空间
看一下最近日志
看一下 docker 容器状态
看一下 k8s pod 日志
帮我按 runbook 排查高 CPU
重启这个服务
```

Planner 应映射：

```text
CPU -> ops.local.processes
磁盘 -> ops.local.disk_usage
内存 -> ops.local.memory_usage
日志 -> ops.local.logs_tail
Docker -> ops.docker.*
K8s -> ops.k8s.*
Runbook -> runbook.*
重启 -> approval required
```

要求：

1. 优先使用结构化 Ops tools。
2. 不要直接生成 shell.exec，除非没有对应 tool。
3. 写操作前生成清晰风险说明。
4. 审批卡片包含 rollback_plan。
5. RunState / RunStep 记录 ops plan。
