==================================================
任务 1：升级 code.fix_test_failure_loop 为多轮修复循环
==================================================

当前 `code.fix_test_failure_loop` 已经能运行测试、解析失败，并输出下一步 patch proposal 指引，但仍偏保守。本阶段要把它升级为真正的 workflow-level 多轮修复循环。

目标流程：

1. run tests
2. parse failure
3. inspect/search/read 相关文件
4. planner 生成 patch proposal
5. 请求审批
6. apply approved patch
7. rerun tests
8. 如果仍失败，进入下一轮
9. 达到 max iterations 后停止
10. 输出最终总结

建议输入：

```go
type CodeFixLoopInput struct {
    Workspace         string `json:"workspace"`
    TestCommand       string `json:"test_command,omitempty"`
    MaxIterations     int    `json:"max_iterations,omitempty"`
    StopOnApproval    bool   `json:"stop_on_approval,omitempty"`
    AutoRerunTests    bool   `json:"auto_rerun_tests,omitempty"`
    FailureContextMax int    `json:"failure_context_max,omitempty"`
}
````

建议输出：

```go
type CodeFixLoopResult struct {
    Iterations      int                   `json:"iterations"`
    FinalPassed     bool                  `json:"final_passed"`
    TestRuns        []RunTestsResult       `json:"test_runs"`
    Failures        []TestFailureParse     `json:"failures,omitempty"`
    ProposedPatches []PatchProposalSummary `json:"proposed_patches,omitempty"`
    AppliedPatches  []PatchApplyResult     `json:"applied_patches,omitempty"`
    StoppedReason   string                `json:"stopped_reason"`
    Summary         string                `json:"summary"`
}
```

要求：

1. 必须有 max iterations，默认 3。
2. 每轮都要落 RunStep。
3. 每个 patch proposal 都是新的 approval snapshot。
4. patch 未审批前不能 apply。
5. 用户拒绝 patch 后 loop 停止。
6. approval 后 workflow resume 应继续 loop，而不是重新从头规划。
7. 如果测试命令本身需要审批，先请求测试命令审批。
8. 不能无限循环。
9. 测试输出和失败信息必须脱敏。
10. loop summary 要说明：

    * 执行了几轮
    * 哪些测试通过/失败
    * 哪些文件被改动
    * 为什么停止

如果完全自动 model-driven patch 生成工作量过大，可以先做：

* loop 状态机
* approval/resume 后 rerun tests
* planner 返回 patch proposal 的结构化接口
* 多轮状态保存

但不能在 README 里夸大“已经能自动修复所有测试失败”。

任务 2：增强 Patch Engine
====================

当前 patch 系统支持 full-file replacement 和基础 unified diff validation/dry-run。本阶段继续增强为更稳的 patch engine。

需要实现或增强：

1. 更完整 unified diff parser
2. partial apply
3. hunk-level conflict detection
4. path escape 防护
5. sensitive path guard
6. expected_sha256 / file hash guard
7. dry-run apply
8. rollback snapshot API
9. changed files summary
10. diff statistics

建议结构：

```go
type UnifiedDiff struct {
    Files []DiffFile `json:"files"`
}

type DiffFile struct {
    OldPath string     `json:"old_path"`
    NewPath string     `json:"new_path"`
    Hunks   []DiffHunk `json:"hunks"`
}

type DiffHunk struct {
    OldStart int        `json:"old_start"`
    OldCount int        `json:"old_count"`
    NewStart int        `json:"new_start"`
    NewCount int        `json:"new_count"`
    Lines    []DiffLine `json:"lines"`
}

type DiffLine struct {
    Kind string `json:"kind"` // context | add | delete
    Text string `json:"text"`
}
```

新增或强化：

```text
code.validate_patch
code.dry_run_patch
code.apply_patch
```

要求：

1. 能解析常见 unified diff。
2. 能检测 hunk context 不匹配。
3. 能检测目标文件 hash mismatch。
4. 能检测路径逃逸。
5. 能识别敏感文件变更。
6. dry-run 不写文件。
7. apply_patch 必须先 validate。
8. apply_patch 必须审批。
9. apply 失败必须 rollback。
10. rollback snapshot 应可追踪。
11. README 明确支持边界，不要宣称完全等价 git apply，除非真的做到。

任务 3：Schema-driven Code Workflow
================================

当前 Planner 已经能启发式选择 code tools。本阶段要把它升级成更稳定的 schema-driven code workflow。

建议新增结构：

```go
type CodeTaskKind string

const (
    CodeTaskInspect CodeTaskKind = "inspect"
    CodeTaskSearch  CodeTaskKind = "search"
    CodeTaskPatch   CodeTaskKind = "patch"
    CodeTaskTest    CodeTaskKind = "test"
    CodeTaskFix     CodeTaskKind = "fix"
    CodeTaskGit     CodeTaskKind = "git"
)

type CodePlan struct {
    Kind             CodeTaskKind  `json:"kind"`
    Workspace        string        `json:"workspace"`
    Goal             string        `json:"goal"`
    Steps            []CodePlanStep `json:"steps"`
    RequiresApproval bool          `json:"requires_approval"`
}

type CodePlanStep struct {
    Tool    string         `json:"tool"`
    Purpose string         `json:"purpose"`
    Input   map[string]any `json:"input"`
}
```

要求：

1. Planner 对代码任务先产出 CodePlan，再逐步转 ToolProposal。
2. CodePlan 不直接执行。
3. 每个 step 仍进入 ToolRouter / Policy / Approval。
4. 修改型任务必须包含 patch proposal step。
5. apply_patch step 必须等 approval。
6. run_tests step 可自动执行，但仍受 Policy 约束。
7. git 写入步骤必须审批。
8. workflow 要能保存 CodePlan 到 RunStep / state，便于 replay。

典型代码任务流程：

```text
inspect_project
search_text/search_symbol
read_file
propose_patch
approval
apply_patch
run_tests
parse_test_failure
continue or summarize
```
