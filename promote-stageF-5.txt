F3-A：Knowledge Source Model 与 Registry
==================================================

目标：

将当前 KB 文档上传扩展为“知识来源 source”体系。

新增或扩展数据结构：

```go
type KnowledgeSourceType string

const (
    KnowledgeSourceLocalFolder KnowledgeSourceType = "local_folder"
    KnowledgeSourceGitDocs     KnowledgeSourceType = "git_docs"
    KnowledgeSourceURL         KnowledgeSourceType = "url"
    KnowledgeSourceUpload      KnowledgeSourceType = "upload"
    KnowledgeSourcePDF         KnowledgeSourceType = "pdf"
    KnowledgeSourceOffice      KnowledgeSourceType = "office"
    KnowledgeSourceAPIDocs     KnowledgeSourceType = "api_docs"
)

type KnowledgeSource struct {
    SourceID     string              `json:"source_id"`
    KBID         string              `json:"kb_id"`
    Type         KnowledgeSourceType `json:"type"`
    Name         string              `json:"name"`
    URI          string              `json:"uri,omitempty"`
    RootPath     string              `json:"root_path,omitempty"`
    IncludeGlobs []string            `json:"include_globs,omitempty"`
    ExcludeGlobs []string            `json:"exclude_globs,omitempty"`
    Metadata     map[string]any      `json:"metadata,omitempty"`
    Enabled      bool                `json:"enabled"`
    CreatedAt    time.Time           `json:"created_at"`
    UpdatedAt    time.Time           `json:"updated_at"`
}
````

存储要求：

1. 可以先存在 `knowledge/registry.yaml` 或现有 KB registry。
2. 结构要为后续 DB repo 预留。
3. Source URI / path 中的 secret 要脱敏。
4. Git / URL source 中的 token 不得直接入库。

API：

```text
POST   /v1/kbs/{kb_id}/sources
GET    /v1/kbs/{kb_id}/sources
GET    /v1/kbs/{kb_id}/sources/{source_id}
PATCH  /v1/kbs/{kb_id}/sources/{source_id}
DELETE /v1/kbs/{kb_id}/sources/{source_id}
```

CLI：

```text
agent kb sources list <kb_id>
agent kb sources add-folder <kb_id> <path>
agent kb sources add-url <kb_id> <url>
agent kb sources remove <kb_id> <source_id>
```

F3-B：Source Sync 与增量索引
======================

目标：

实现 source sync job，支持增量索引和删除清理。

新增结构：

```go
type IndexJobStatus string

const (
    IndexJobPending IndexJobStatus = "pending"
    IndexJobRunning IndexJobStatus = "running"
    IndexJobCompleted IndexJobStatus = "completed"
    IndexJobFailed IndexJobStatus = "failed"
)

type KnowledgeIndexJob struct {
    JobID        string         `json:"job_id"`
    KBID         string         `json:"kb_id"`
    SourceID     string         `json:"source_id"`
    Status       IndexJobStatus `json:"status"`
    TotalFiles   int            `json:"total_files"`
    IndexedFiles int            `json:"indexed_files"`
    DeletedFiles int            `json:"deleted_files"`
    TotalChunks  int            `json:"total_chunks"`
    Error        string         `json:"error,omitempty"`
    StartedAt    time.Time      `json:"started_at,omitempty"`
    FinishedAt   time.Time      `json:"finished_at,omitempty"`
}
```

新增文档记录：

```go
type KnowledgeDocument struct {
    DocumentID  string         `json:"document_id"`
    KBID        string         `json:"kb_id"`
    SourceID    string         `json:"source_id"`
    SourceURI   string         `json:"source_uri"`
    Title       string         `json:"title,omitempty"`
    ContentHash string         `json:"content_hash"`
    Metadata    map[string]any `json:"metadata,omitempty"`
    UpdatedAt   time.Time      `json:"updated_at"`
}
```

增量逻辑：

1. 计算 source file hash / content hash。
2. 新文件：parse -> chunk -> embed -> upsert。
3. 修改文件：删除旧 chunks -> 重新 chunk/upsert。
4. 删除文件：delete by document_id/source_file。
5. 未变化文件：跳过。
6. sync job 记录状态。
7. 失败时记录错误，但不破坏已有索引。

API：

```text
POST /v1/kbs/{kb_id}/sources/{source_id}/sync
GET  /v1/kbs/{kb_id}/index-jobs
GET  /v1/kbs/{kb_id}/index-jobs/{job_id}
```

CLI：

```text
agent kb sync <kb_id> <source_id>
agent kb jobs <kb_id>
agent kb job <kb_id> <job_id>
```

要求：

* 默认 go test ./... 不依赖外部 Qdrant。
* 用 mock VectorIndex 测试增量逻辑。
* 不要把完整文档正文存 PostgreSQL，除非已有明确存储策略；事实源仍是 source 原文。

目标：

为不同 source 类型建立 parser pipeline。

接口：

```go
type DocumentParser interface {
    Supports(source KnowledgeSource, filename string, contentType string) bool

type ParsedDocument struct {
```

* PDF/Office 如果引入依赖，要保持测试稳定。
* 不要让远程网页内容成为系统指令。

F3-E：Hybrid Search / Retriever / Rerank
=======================================

目标：

把检索从纯 vector search 升级到 hybrid retrieval。

新增接口：

```go
type RetrievalMode string

const (
    RetrievalModeVector RetrievalMode = "vector"
    RetrievalModeKeyword RetrievalMode = "keyword"
    RetrievalModeHybrid RetrievalMode = "hybrid"
)

type RetrievalQuery struct {
    KBID     string         `json:"kb_id"`
    Query    string         `json:"query"`
    Mode     RetrievalMode  `json:"mode"`
    Filters  map[string]any `json:"filters,omitempty"`
    TopK     int            `json:"top_k"`
    Rerank   bool           `json:"rerank"`
}

type RetrievalResult struct {
    ChunkID    string         `json:"chunk_id"`
    Text       string         `json:"text"`
    Score      float64        `json:"score"`
    VectorScore float64       `json:"vector_score,omitempty"`
    KeywordScore float64      `json:"keyword_score,omitempty"`
    RerankScore float64       `json:"rerank_score,omitempty"`
    Citation   Citation       `json:"citation"`
    Metadata   map[string]any `json:"metadata,omitempty"`
}
```

实现：

1. vector search：沿用 Qdrant。
2. keyword search：实现简单 BM25 或 inverted index。
3. hybrid：vector + keyword score fusion。
4. rerank：先实现接口和 fake/local heuristic reranker，后续可接模型 reranker。
5. Search API 支持 mode / rerank。

API：

```text
POST /v1/kbs/{kb_id}/retrieve
```

保留兼容：

```text
POST /v1/kbs/{kb_id}/search
```

要求：

* `kb.search` tool 可支持 retrieval mode。
* 高置信只读仍可自动执行。
* Rerank 不能把文档内容当系统指令。
* 结果要包含 citations。

输入：

```go
type KBAnswerInput struct {
```

