任务 1：升级 SandboxRunner 抽象
==================================================

请在现有 SandboxRunner / LocalRestrictedRunner 基础上升级抽象。

目标：

1. 保留现有 LocalRestrictedRunner
2. 新增更强 profile / runner 抽象
3. 为 Linux 场景引入真正更强的隔离路线

建议抽象：

```go
type SandboxProfile string

const (
    SandboxProfileBestEffortLocal   SandboxProfile = "best_effort_local"
    SandboxProfileRestricted        SandboxProfile = "restricted"
    SandboxProfileTrustedLocal      SandboxProfile = "trusted_local"
    SandboxProfileLinuxIsolated     SandboxProfile = "linux_isolated"
)

type ExecutionProfile struct {
    SandboxProfile    SandboxProfile
    AllowedReadPaths  []string
    AllowedWritePaths []string
    NetworkEnabled    bool
    AllowedHosts      []string
    AllowedEnv        []string
    TimeoutSeconds    int
    MaxOutputBytes    int64
    MaxProcesses      int
    MemoryLimitMB     int
    CPULimitPercent   int
    WorkingDirectory  string
}

type SandboxRequest struct {
    Command string
    Args    []string
    CWD     string
    Env     map[string]string
    Stdin   []byte
    Profile ExecutionProfile
}

type SandboxResult struct {
    Stdout     string
    Stderr     string
    ExitCode   int
    DurationMS int64
    Warnings   []string
    Metadata   map[string]any
}

type SandboxRunner interface {
    Run(ctx context.Context, req SandboxRequest) (*SandboxResult, error)
    Name() string
    Supports(profile SandboxProfile) bool
}
````

要求：

* 不要删除现有 best_effort 路径
* 要让 SkillRunner 能根据 manifest.sandbox.profile 选择不同 runner
* 没有合适 runner 时，行为要明确：

  * 拒绝执行
  * 或降级为 best_effort 并强制审批
* 行为必须可测试

任务 2：实现 Linux 优先的更强隔离 Runner MVP
================================

本阶段不要求做容器编排平台，但必须实现一个“比 LocalRestrictedRunner 更强”的 Linux runner。

建议命名：

* LinuxIsolatedRunner
* SeccompNamespaceRunner
* 或等价清晰命名

最低目标：

1. Linux 平台启用
2. 支持 timeout
3. 支持 max output bytes
4. 支持 env allowlist
5. 支持 cwd 限制
6. 支持 read/write 路径限制（哪怕是 best-effort + wrapper，也要比现有更明确）
7. 支持尽量关闭网络或限制网络
8. 支持最小权限运行

建议优先考虑这些技术路线（按可行性择一或组合）：

* `exec.Cmd` + `SysProcAttr` 最小权限运行
* Linux namespace（如 user/mount/net/pid namespace）
* seccomp profile（如可落地）
* cgroup/resource caps
* 受限 mount / chroot / pivot_root（如果你能做得稳）
* 限制子进程数 / fd / memory / cpu
* 禁网或 allowlist（若做不到真正 OS 级 host allowlist，必须如实记录为 best-effort）

要求：

* 不要伪装成“强隔离”但实际上只是 timeout+env
* 如果某些隔离只在 Linux 可用，必须：

  * runtime 检测平台
  * 非 Linux 明确 fallback
  * README 明确平台差异
* 如果你做的是 Linux-only MVP，也可以；但要清晰、不夸大

任务 3：将 manifest.sandbox.profile 与 runner 选择真正接起来
================================================

manifest 现在已经有：

现在要把这些真正用在 runner 选择和执行期 enforcement 中。

要求：

1. `best_effort_local`

   * 继续使用 LocalRestrictedRunner
2. `restricted`

   * 优先尝试更强 runner
   * 若当前平台不支持，行为必须明确：

     * 拒绝执行
     * 或降级并强制审批
3. `trusted_local`

   * 允许放宽本地限制，但仍不能绕过 policy
   * 不能自动提升到“无边界执行”
4. `linux_isolated`（如果你新增）

   * 只在 Linux 支持
   * 非 Linux 拒绝或明确降级

manifest 与 runtime 的不一致处理要求：

* permissions.network.enabled=false，但 runner 实际无法禁网：

  * 记录 warning
  * 标记 best_effort
  * README 写清楚
* permissions.write 非空但实际 runner 无法 enforce：

  * 不能伪装成功
  * 至少在 validate/run 前给出 warning 或 require approval
* 不允许 manifest 声明比实际执行能力更安全，而系统悄悄放宽

任务 5：权限与 policy 的一致性收紧
======================

进一步强化 manifest 的 consistency 校验。

要求：

1. 如果 permissions 含 write path，但 effects 不含 write/config/code modify 类效果：

   * validate 失败，或至少 warning + force approval
2. 如果 network.enabled=true 但 effects 没有 network.*：

   * validate 失败，或至少 warning + force approval
3. 如果 tools.allow_shell=true，但 effects 没有 shell/process 相关效果：

   * validate 失败，或至少 warning + force approval
4. 如果 sandbox.profile=trusted_local：

   * 不等于自动允许危险效果
   * 仍必须服从 policy
5. SkillRunner 在执行前应拿到“最终安全决议”，而不是只看 manifest

也就是说：

* manifest 只是声明
* effect inference / policy 是系统主判定
* sandbox 是执行期约束
* 三者不能彼此冲突
