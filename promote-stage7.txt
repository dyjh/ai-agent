任务 2：把 Workflow 从“单步链路”增强为 Step-based Loop
==========================================

当前 Workflow 已能处理单次 tool 调用 + approval resume。请增强为显式 step loop。

目标：

* 支持一个 run 内有多个 step
* 每个 step 可能是：

  * build_context
  * plan
  * propose_tool
  * infer_effect
  * decide_policy
  * request_approval
  * execute_tool
  * summarize
  * finalize
* 一个 run 可以多次经历：
  propose_tool -> infer -> policy -> execute -> observe -> continue?
* 直到 planner 判断完成，或进入失败/取消

请定义清晰的 Step 模型，例如：

```go
type RunStepType string
type RunStepStatus string

type RunStep struct {
    StepID       string
    RunID        string
    Index        int
    Type         RunStepType
    Status       RunStepStatus
    Proposal     *core.ToolProposal
    Inference    *core.EffectInferenceResult
    Policy       *core.PolicyDecision
    ApprovalID   string
    ToolResult   *core.ToolResult
    Summary      string
    Error        string
    CreatedAt    time.Time
    UpdatedAt    time.Time
}
```

要求：

* step 必须可序列化
* 每个 step 要落 JSONL / agent_events / run state repo
* 当前阶段不需要无限复杂自动规划
* 但至少要支持：

  * tool 结果回来后，planner 可以决定是否再走一步
  * 限制最大 step 数，防止无限循环
  * 对每一步都重新走 effect/policy/approval 判定

建议加入：

```go
MaxWorkflowSteps int
```

默认例如 6 或 8。

如果超过上限：

* 终止 run
* 给用户清晰说明
* 记录 run.failed 或 run.cancelled

任务 4：强化 Resume 语义
=================

当前 Resume 已可用，本阶段要加强它的 durability 和幂等性。

要求：

1. Resume 只能从 durable state 恢复
2. approved=true 时：

   * 只能执行当时 approval 绑定的 input_snapshot
   * 执行前校验 hash / snapshot integrity（如果项目已有 hash，要复用）
3. approved=false 时：

   * 不执行 executor
   * 写入 rejection step/event
   * 由 planner/summarizer 输出一个明确的 assistant message
4. 同一个 approval 不得重复执行
5. runID / approvalID 不匹配必须报错
6. 已 completed / failed / cancelled 的 run 不可再 resume

另外请增加：

* 服务重启后还能 resume 审批中的 run
* 如果 state 缺失但 approval 还在，给出清晰错误，而不是 silent fail

任务 5：增加 Run 查询与恢复管理 API
=======================

请新增一组 run 管理接口，用于调试、可观测性和恢复：

   * 作为 approval API 的补充入口

   * 如果当前处于 paused_for_approval，可直接取消

* 错误响应统一

  * `agent runs list`
