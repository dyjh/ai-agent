任务 1：增加 Skill ZIP 上传与安装流程
==================================================

请在现有 skill 管理基础上，增加 ZIP 上传与安装能力。

目标：

1. 支持通过 API 上传 zip 包
2. 解压到受控目录，例如：
   - skills/packages/<skill-id>/<version>/
   - 或等价安全目录
3. 校验 skill.yaml
4. 更新 registry
5. 成功后可被 manager 发现和运行

新增能力建议：

- `POST /v1/skills/upload-zip`
- 或复用现有 upload，但必须明确支持 multipart zip

要求：

1. Zip 解压前做安全检查：
   - 防止 path traversal
   - 防止 zip slip
   - 限制文件数量
   - 限制总解压大小
   - 限制单文件大小
2. 解压后必须找到：
   - skill.yaml
3. manifest 校验通过后才注册
4. 记录：
   - skill id
   - version
   - source type = zip
   - checksum（建议 sha256）
   - installed_at
5. 同一个 skill id + version 重复安装的行为要明确：
   - 可以拒绝
   - 或 allow overwrite with force flag
   - 但必须一致、可测试
6. 上传失败时要清理临时目录
7. 不要在上传阶段直接运行 skill

建议新增：

```go
type SkillPackageInfo struct {
    SkillID      string
    Version      string
    SourceType   string
    PackagePath  string
    Checksum     string
    InstalledAt  time.Time
}
````

任务 2：扩展 Skill manifest，增加 sandbox / isolation / permissions
===========================================================

请在现有 skill.yaml 基础上扩展以下语义。

建议 manifest：

```yaml
id: cpu_analyzer
name: CPU Analyzer
version: 1.0.0
description: Analyze local CPU and process information.

runtime:
  type: executable # executable | script
  command: ./bin/cpu_analyzer
  args: []
  cwd: .
  input_mode: json_stdin
  output_mode: json_stdout
  timeout_seconds: 30

effects:
  - process.read
  - system.metrics.read

approval:
  default: auto

permissions:
  filesystem:
    read:
      - ./workspace
      - ./logs
    write: []
  network:
    enabled: false
    allow_hosts: []
  env:
    allow:
      - LANG
  tools:
    allow_shell: false
    allow_mcp: false
  process:
    max_runtime_seconds: 30
    max_output_bytes: 1048576

sandbox:
  profile: restricted # restricted | trusted_local | best_effort_local
```

要求：

1. 兼容旧 manifest，给出合理默认值
2. 缺失 permissions 时采用最保守默认：

   * no network
   * no write
   * minimal env
3. `effects` 和 `permissions` 不是一回事：

   * effects 决定 Policy / Approval
   * permissions 决定执行期允许的行为
4. manifest 校验必须覆盖：

   * sandbox.profile 合法值
   * path 列表格式
   * timeout / output 限制
   * network allow_hosts 格式
5. README 和 Swagger 示例需要同步更新 manifest 结构

任务 3：引入 SandboxRunner / ExecutionProfile 抽象
===========================================

请不要把 runner 继续做成“直接 exec 就完了”。

新增一个抽象层，例如：

```go
type ExecutionProfile struct {
    SandboxProfile    string
    AllowedReadPaths  []string
    AllowedWritePaths []string
    NetworkEnabled    bool
    AllowedHosts      []string
    AllowedEnv        []string
    TimeoutSeconds    int
    MaxOutputBytes    int64
}

type SandboxRunner interface {
    Run(ctx context.Context, req SandboxRequest) (*SandboxResult, error)
}
```

其中：

```go
type SandboxRequest struct {
    Command     string
    Args        []string
    CWD         string
    Env         map[string]string
    Stdin       []byte
    Profile     ExecutionProfile
}
```

要求：

1. 当前阶段不要求做重量级容器沙箱，但要有统一抽象
2. 默认先实现一个 `LocalRestrictedRunner`
3. `LocalRestrictedRunner` 至少做到：

   * timeout
   * max output bytes
   * env allowlist
   * cwd 限制
   * 可执行文件路径校验
   * 受控 read/write path 检查（尽可能在运行前校验 + 明确限制）
4. 如果某些文件系统/网络限制在本阶段不能完全做到 OS 级隔离：

   * 必须在 README 明确说明
   * 代码中保留清晰的 SandboxRunner 扩展点
5. 不要假装做到了强隔离而实际只是普通 exec

任务 4：将 permissions enforcement 接入 SkillRunner
=============================================

请把 manifest.permissions 真正用起来，而不只是解析后放着。

要求：

1. SkillRunner 在执行前：

   * 校验 runtime.command 路径是否在允许范围
   * 校验 cwd
   * 组装受限 env
   * 组装 ExecutionProfile
2. 如果 manifest 声明：

   * network.enabled=false
   * 本阶段若无法 OS 级完全阻断，必须：

     * 记录 warning

   * 拒绝执行
4. 如果 skill 需要 shell / mcp / network / write：

   * effects 和 policy 必须匹配
   * 不允许 manifest.permissions 比 policy 更宽松
5. 执行结果仍要经过脱敏与结构化输出

任务 6：补更细粒度的 Skill Policy / Approval 映射
======================================

请进一步强化 Skill 的 effect -> approval 映射。

要求：

1. 如果 permissions 包含写路径或 network.enabled=true，但 effects 仍声明为纯 read：

   * 视为 manifest 不一致
   * validate 失败，或至少 warning + require approval
2. 如果 sandbox.profile=trusted_local：

   * 不能自动提升权限
   * 仍必须服从 policy
3. `approval.default=require` 仍优先要求审批
4. unknown effects 仍默认审批
5. 只读高置信 skill 仍可自动执行

建议增加 manifest consistency 校验：

* permissions.network.enabled=true 但 effects 没有 network.*
* write paths 非空但 effects 没有 fs.write / code.modify / config.modify
* tools.allow_shell=true 但 effects 没有与 shell 行为相关声明

不要求过度智能，但要尽可能防止明显不一致。
